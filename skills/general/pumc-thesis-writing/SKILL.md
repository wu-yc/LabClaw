# 协和医学院博士论文写作规范

## 一、总体结构：部分制（铁律）

协和官方模板（2025版）使用"第1章"作为一级结构。但医学类院所的博士论文普遍采用**部分制**，这是协和医学论文的通行惯例。以下以部分制为准，章节制同学参照官方模板即可。

```
封面 / 目录 / 缩略词表 / 中文摘要 / English Abstract
/ 引言或前言（全文级）
/ 第一部分 / 第二部分 / ... / 第N部分
/ 结论（全文级）/ 文献综述 / 参考文献 / 致谢
```

> 注：上述 `/` 仅为skill内部示意分隔，论文正文中不出现。

## 二、每个"部分"的内部结构（铁律）

每个"部分"是一篇独立的研究论文，节奏固定：

```
中文摘要（研究背景 / 研究目的 / 研究方法 / 研究结果 / 研究结论）
English Abstract
前言
材料和方法
结果
讨论
结论
参考文献（本部分独立）
```

建库/建系统类工作同样遵循此格式，"材料和方法"变为"系统构建方法"，"结果"变为"系统实现与验证结果"。

一个部分如果包含多项独立子研究，可拆为(一)(二)(三)，每个子研究各自完整走一遍上述流程。

## 三、标题、段落与编号格式

### 3.1 标题层级

采用阿拉伯数字层级编号，各层用圆点分隔，末位不加点。标题**独占一行**。

```
1  一级/章标题（黑体三号，居中，段前24磅段后18磅）
1.1  二级标题（黑体小三号，顶左无缩进，段前24磅段后6磅）
1.1.1  三级标题（黑体四号，顶左无缩进，段前12磅段后6磅）
1.1.1.1  四级标题（宋体小四号，左对齐缩进2字符，段前12磅段后6磅）
```
注意：只有四级标题需要缩进2字符，一至三级均**顶格左对齐**（官方用词"顶左"）。

标题以三级为宜，最多四级。**标题结尾不加标点符号。** 不混用"第一节"、"（一）"等中文编号与阿拉伯数字。

### 3.2 禁止行内加粗标题

**`**加粗标题。** 正文接续...`** 是英文期刊（Nature/Science）Methods部分的写法。协和博士论文不用。

- ❌ `**构建目的。** OmniQA旨在评估...`
- ✅ 把"构建目的"作为编号小标题独占一行，下方跟正文段落

### 3.3 列表与枚举

正文中列举事项用 `（1）（2）（3）` 行内编号，不用 `- ` markdown无序列表。

- ❌ `- 结构化引导的贡献 = GRACE w/o ET − Linear ReAct`
- ✅ `（1）结构化引导的贡献，即GRACE w/o ET与Linear ReAct之间的性能差值；（2）...`

### 3.4 段落基本要求

- 每段首行缩进2字符
- 段落内不再设子标题，直接叙述
- 公式/数学表达式单独成行居中，编号右对齐
- 行内公式用 `$...$`（如 `$\alpha = 0.5$`），独立公式用 `$$...$$`
- `pumc-create-docx` 会将 LaTeX 公式自动转为 Word 可编辑公式对象
- 简单变量赋值（如 k = 60、α = 0.5）也可直接写纯文本 + Unicode，不必用 LaTeX

## 四、前言写法

**连贯叙述体**，不用分节标题。1-3页。第三人称"本研究"。

隐含结构（不显式标注）：
1. 研究领域背景与重要性（1-2段）
2. 当前研究现状与已有工作（1-2段）
3. 现有研究的不足（1段）
4. 本部分的研究目的与策略（1段）

段落间靠语义承接自然过渡，不依赖"此外""同时"等连接词开头。

## 五、结果写法

### 5.1 基本原则
- 使用编号小节，每节围绕一个核心发现展开
- 先用文字陈述发现，再引用图表佐证（"如图1-1所示"、"结果见表1-2"）
- **只陈述客观结果，不做深入解读**（解读放讨论）
- 数据要精确：具体数字、百分比、p值、置信区间

### 5.2 图表引用规范
- 引用格式统一：`如图X-Y所示`、`结果见表X-Y`、`表X-Y列出了...`
- 每个图表在正文中至少被引用一次
- 图表编号按出现顺序递增，不跳号

### 5.3 数据描述示例

- ❌ `DPER取得了显著的效果`（空话）
- ✅ `使用Gemini嵌入模型时，DPER达到88.5%的Hit@1（MRR 92.7%，NDCG@5 93.7%），较Doc-only策略提升17.4个百分点（表1-6）`

## 六、讨论写法

### 6.1 结构框架
讨论以连贯叙述为主，可设少量小标题。典型结构：

1. **首段**：用2-3句概括本研究的核心发现（不是重复结果，是提炼核心结论）
2. **中间段**（每段围绕一个关键发现）：
   - 先重述该发现的一句话结论
   - 与已有文献对比（一致或矛盾之处）
   - 解释可能的机制或原因
3. **局限性段**（1段）：客观陈述不足，不回避
4. **末段**：总结本研究的意义，展望未来方向

### 6.2 常见错误
- ❌ 简单重复结果节的数据（讨论要有**解读和对比**）
- ❌ 引入结果节未提及的新数据
- ❌ 把局限性段写成"但本研究仍具有重要意义"的自我辩护

## 七、结论写法

结论是对本部分核心发现的提炼，不是各章小结的简单拼接。

- 精炼概括本部分的核心观点和创造性成果
- 可用编号（1、2、3）逐条列出，也可连贯叙述
- **严格区分本人成果与导师及他人的贡献**（协和官方要求）
- **避免"首次""领先""填补空白"等词语**，除非有充分证据支撑（协和官方要求）
- 可简要说明研究的局限性和未来方向
- 不引入正文未涉及的新内容
- 篇幅控制在1页以内

## 八、摘要写法

### 中文摘要
博士论文中文摘要 **1500字左右**。五段式结构：
```
研究背景：[2-3句]
研究目的：[1-2句]
研究方法：[详细]
研究结果：[详细，含关键数据]
研究结论：[2-3句]
关键词：[3-5个，分号隔开，最后不加标点]
```

### English Abstract
结构与中文严格对应。KEYWORDS全部大写，每个关键词首字母大写，分号隔开，最后不加标点。

## 九、图表与缩略词规范

### 图表编号与格式
- 编号：图1-1、表1-1（X=部分号，Y=序号）；含子研究：图2-1-1
- **图序、图题**置于图下方，楷体五号字居中，编号与题名间空1字符，**题后不加标点**
- **表序、表题**置于表上方，楷体五号字居中，编号与题名间空1字符，**题后不加标点**
- 表采用国际通行的**三线表**
- 每个图表在正文中至少被引用一次

### 缩略词
- 首次出现：**中文全称（英文全称，英文缩写）**
- 例：`差异表达分析（Differential Expression Analysis, DEA）`
- 缩略词表按英文缩写字母顺序，三列：英文缩写 | 英文全称 | 中文全称

## 十、去AI味（核心规则）

AI生成的中文学术文本有强烈的可识别指纹。以下规则从源头遵守。

### 9.1 禁用符号

| 符号 | 规则 |
|------|------|
| `——`（破折号） | 全文不超过2-3处。用括号、逗号或改写句子代替 |
| `→`（箭头） | 正文禁止。只能出现在图表内部。流程用自然语言表达 |
| emoji（✅❌🔴等） | 正文禁止 |
| `*`（markdown语法） | 正文禁止 |

### 9.2 AI标志词黑名单

| 黑名单 | 替换 |
|---|---|
| 值得注意的是 | 删除，直接说 |
| 综上所述 / 总而言之 | 换具体结论 |
| 在当今...背景下 / 随着...的快速发展 | 直接切入事实 |
| 不可忽视 / 至关重要 / 极其重要 | 用数据说话 |
| 提供了新的思路/视角 | 说清楚什么思路 |
| 具有重要的理论和实践意义 | 拆开具体说 |
| 日益增长/日益加深 | 用具体数据 |
| 为...奠定了坚实基础 | 说清楚奠定了什么 |
| 研究表明（无来源） | "Zhang等[12]发现..." |
| 引起了广泛关注 | 用具体事件代替 |
| 旨在探索/揭示/阐明（堆叠） | 一个动词直说："本研究构建了..." |
| 近年来（段首高频） | 用具体年份 |

> 上述词汇单次出现正常。AI的问题是**高密度重复**。关注频率，非零容忍。

### 9.3 中文标点与混排

- **顿号（、）**：并列的词或短语
- **逗号（，）**：句内停顿
- **分号（；）**：并列的分句
- AI常在该用顿号的地方用逗号，需注意区分
- 中英文之间加半角空格：`共 270 个数据库`

### 9.4 句式去机械化

AI连接词密度约7.2个/千字，人类约4.1个/千字。

- **变化句长**：3-4个长句后插一个短句收束
- **变化句式**：不全用"A通过B实现了C"。混用主动/被动/因果/转折
- **控制连接词**：不要每段都以"此外/同时/另外/因此"开头
- **段落长短不一**：有的5-6句，有的2-3句
- **偶尔短句开段**：`这一问题并非新近出现。`

### 9.5 去空洞

每句必须有实质内容。

- ❌ `该方法取得了显著的效果，为后续研究提供了重要参考。`
- ✅ `DPER在732函数基准上达到88.5%的Hit@1，较单通道策略提升17.4个百分点。`

### 9.6 过渡去模板化

- ❌ 每段靠"此外""然而""因此"串联
- ✅ 用上段末句概念作为下段首句起点（语义勾连）
- ✅ 直接陈述新事实，读者自然理解逻辑关系

**对比示例：**

AI风格：
> 此外，现有系统在工具检索方面也存在不足。值得注意的是，大多数系统依赖单通道语义检索，这在面对不同类型查询时表现出显著的局限性。

人类风格：
> 工具检索精度同样制约着现有系统的实用性。多数系统依赖基于文档描述的单通道语义检索，面对技术性查询与自然语言查询时呈现出互补盲区。

## 十一、叙事风格铁律

### 必须做的
- "本研究"第三人称客观叙事
- 术语首次出现给全称+缩写
- 数据精确到具体数值
- 比较用语客观（"显著高于"、"差异具有统计学意义"）

### 绝对不做的
- 不用"我们"
- 不用技术文档风格（不列API端点、代码配置、Docker命令）
- 不在前言中展开方法和结果
- 不在结果中做深入解读（放讨论）
- 不在讨论中引入新数据
- 不在无充分证据时使用"首次""领先""填补空白"（协和官方禁令）

## 十二、参考文献格式

> 协和官方规范（2025版）规定参考文献"集中著录于正文之后，不分章节著录"。但部分制论文的实际做法中，各部分末尾各有独立参考文献列表（如魏之瑶、赵狄铭论文），应以导师要求为准。

### 正文引用
- 用上标方括号：`[1]`、`[2]`
- 同处引多篇：`[1,3]`
- 连续序号：`[2-5]`
- 多次引同一文献不同页码：`[2]260`、`[2]326-329`

### 文后著录
遵循 GB/T 7714-2015。期刊析出文献格式：
```
[1] 作者. 题名[J]. 期刊名, 年, 卷(期): 起页-止页.
```
- 编著者3人及以下全部著录；3人以上著录前3人，后加",等"或", et al."（不用斜体）
- 文后参考文献中符号使用英文半角
- 所有被引用的文献必须列入，未引用的不得出现

## 十三、Markdown 写作约定（配合 docx 转换）

以下约定确保 `.md` 文件能被 `pumc-create-docx` 脚本正确转换为协和格式的 `.docx`。

### 标题层级映射

| Markdown | 转换为 | 示例 |
|----------|--------|------|
| `# 标题` | 章/部分标题（黑体三号居中） | `# 前言` |
| `## 1.1 标题` | 二级标题（黑体小三号顶左） | `## 1.1 系统总体架构设计` |
| `### 1.1.1 标题` | 三级标题（黑体四号顶左） | `### 1.1.1 设计目标` |
| `#### 1.1.1.1 标题` | 四级标题（宋体小四号缩进） | `#### 1.1.1.1 核心约束` |

注意：节编号（1.1、1.1.1等）写在标题文字中，脚本不自动编号。

### 图题与表题

独占一行，以 `图` 或 `表` + 空格 + 编号开头，脚本自动识别为楷体五号居中：
```
图 1-1 OmniMind 功能架构总览
表 1-2 OmniAtlas 领域覆盖详表
```

### 图片占位

使用标准 markdown 图片语法，脚本在 docx 中生成 `[此处插入图片]` 占位符 + 图题：
```
![图 1-1 OmniMind 功能架构总览](figures/fig1-1.png)
```

### 关键词

独占一行，以 `关键词：` 或 `KEYWORDS:` 开头：
```
关键词：多智能体协同；生物信息学；工具检索；大语言模型
```

### 参考文献条目

每条独占一行，以 `[N] ` 开头（N为编号），脚本自动识别为宋体五号行距16磅：
```
[1] Stephens ZD, et al. Big Data: Astronomical or Genomical? PLoS Biol, 2015.
[2] He FC, et al. π-HuB: the proteomic navigator. Nature, 2024.
```

### 正文中的引用标记

`[1]`、`[2-5]`、`[1,3]` 等引用标记在正文中会自动转为上标，无需特殊处理。

### 表格

md 表格自动转为 Word 三线表（顶线 + 表头下线 + 底线，无竖线），表内文字宋体五号。注意：
- 表格单元格中暂不支持 `$...$` 公式，需转 docx 后在 Word 中手动插入
- 居中对齐列使用 `:---:` 语法
- 表题独占一行写在表格上方（`表 X-Y 标题`），脚本自动识别为楷体五号居中

### 应避免的 Markdown 语法

以下语法在 md 中可读性好但转 docx 效果不佳：
- `` ``` `` 代码块：转为 Courier 等宽字体，不符合协和正文格式。论文正文中技术内容应翻译为学术语言，不保留代码片段
- `> ` 引用块：转为普通正文段落，失去视觉区分。需要强调的内容直接在正文中表述

### 会被自动剥离的 markdown 语法

`**加粗**`、`*斜体*`、`` `代码` ``、`^[1]^` 中的格式标记在转换时会被自动去除，只保留纯文本。因此 markdown 中可以正常使用这些标记辅助阅读，不影响最终 docx 输出。

## 十四、质量检查清单

### 格式
- [ ] "本研究"而非"我们"
- [ ] 术语首次出现给全称+缩写
- [ ] 前言为连贯叙述体
- [ ] 小标题独占一行，不用行内加粗标题
- [ ] **标题结尾不加标点**
- [ ] 编号用阿拉伯数字层级（1 / 1.1 / 1.1.1），最多四级
- [ ] 列举用（1）（2）（3），不用markdown列表
- [ ] 图表编号符合图X-Y格式，**图题/表题后不加标点**
- [ ] 关键词3-5个，分号隔开，**最后不加标点**
- [ ] 正文引用用上标[N]，连续序号用[2-5]

### 系统/工具类论文专项
- [ ] 无API端点、代码配置、Docker命令
- [ ] 技术实现翻译为学术语言
- [ ] 系统规模数据以表格呈现
- [ ] 用"方法"而非"实现"描述设计

### 去AI味
- [ ] 破折号（——）不超过2-3处
- [ ] 无 → 箭头
- [ ] 无"值得注意的是""综上所述"等标志词
- [ ] 无"随着...快速发展"式开头
- [ ] 连接词密度正常（不每段都以"此外"开头）
- [ ] 段落长短有变化
- [ ] 长短句交替
- [ ] 无空洞表达
- [ ] 朗读测试：不像AI在说话
